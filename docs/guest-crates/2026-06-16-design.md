# Guest Crates — Design Spec

**Status:** Approved for phased implementation (pending user sign-off)  
**Date:** 2026-06-16  
**Prod constraint:** No breaking changes to existing personal collections or live URLs.

---

## Goal

Import a friend's **public Discogs collection** into an isolated crate, demo MyVinyl (insights, enrichment, play, PDF, labels), then **hand it off** via a logical URL so they can sign up and own their enriched library.

**Not in scope for v1:** Discogs OAuth, multi-tenant admin, public unauthenticated full-library browse without invite token.

---

## User flow (target)

```
1. You get friend's Discogs username (public — no password)
2. Import → new Guest Crate (never touches your Personal Crate)
3. Enrich + demo (insights, play, PDF, labels)
4. Copy invite link → https://myvinyl-nine.vercel.app/c/pete?invite=<token>
5. Friend opens link → preview → signs up → claims crate
6. Ownership transfers → their /collection is fully theirs (enrichment included)
```

---

## Non-breaking guarantees

| Guarantee | How |
|-----------|-----|
| Existing `/collection`, `/insights`, `/play`, `/labels` work unchanged | Personal crate remains default; no new required URL segments for current users |
| Existing `records` rows stay valid | Additive columns + backfill; no column drops or renames |
| Old clients (cached bundles) safe during rollout | `collection_id` nullable until backfill completes; fetch fallback `user_id` only until Phase 1 app ships |
| Guest data never mixes with personal | Hard `collection_id` filter on all reads/writes once Phase 1 ships |
| Prod deploy order | **DB migration → backfill → app** (never app before backfill) |

## Schema (additive only)

### New table: `collections`

```sql
create table public.collections (
  id                uuid primary key default gen_random_uuid(),
  owner_user_id     uuid not null references auth.users(id) on delete cascade,
  imported_by_user_id uuid references auth.users(id) on delete set null,
  kind              text not null check (kind in ('personal', 'guest', 'pending_claim')),
  name              text not null,
  slug              text not null,
  discogs_username  text,
  record_count      int not null default 0,
  claim_token_hash  text,           -- Phase 2: bcrypt/argon hash of invite token
  claimed_at        timestamptz,
  created_at        timestamptz not null default now(),
  updated_at        timestamptz not null default now()
);

-- One personal crate per user
create unique index collections_one_personal_per_user
  on public.collections (owner_user_id)
  where kind = 'personal';

-- Slugs unique per owner (you can have /crates/pete and /crates/pete-2)
create unique index collections_owner_slug
  on public.collections (owner_user_id, slug);

create index collections_owner_kind on public.collections (owner_user_id, kind);
```

### Alter `records` (nullable first)

```sql
alter table public.records
  add column if not exists collection_id uuid references public.collections(id) on delete cascade;

create index if not exists records_collection_id on public.records (collection_id);
```

**Backfill (required before Phase 1 app deploy):**

1. For each distinct `user_id` in `records`, insert `collections` row: `kind='personal'`, `slug='my-crate'`, `name='My Crate'`.
2. `UPDATE records SET collection_id = <personal_id> WHERE user_id = ? AND collection_id IS NULL`.

After backfill, app treats `collection_id IS NULL` as legacy edge case → lazy-fix on read.

### Limits

| Limit | Value |
|-------|-------|
| Records per collection (vinyl) | **1,500** |
| Active guest crates per account | **5** |
| Slug max length | 64 |

CD/skipped rows do not count toward cap (existing `isCdFormat` filter).

## RLS (Supabase)

```sql
-- collections: owner only
owner_user_id = auth.uid()

-- records: via collection ownership
collection_id in (
  select id from collections where owner_user_id = auth.uid()
)
-- Legacy fallback during migration window:
or (collection_id is null and user_id = auth.uid())
```

Phase 2 adds policies for `pending_claim` preview reads (token-verified RPC, not open RLS).

## URL map

| Phase | URL | Purpose |
|-------|-----|---------|
| Now | `/collection` | Personal crate (unchanged) |
| 1 | `/crates` | Your guest crate list |
| 1 | `/crates/:slug` | View/demo a guest crate |
| 2 | `/c/:slug?invite=` | Friend landing + claim |
| 2 | After claim | Friend uses `/collection` (personal) |

Existing paths keep working. Guest routes are **additive**.

---

## UI principles

- **Subtle:** Crate switcher is a quiet dropdown in the collection hero — not a second nav bar
- **Premium:** Same typography/spacing as existing filters; no loud badges
- **Logical:** Personal crate is always first; guest crates labeled with Discogs username
- **Safe:** Persistent slim banner on guest view: *"Pete's Crate · Demo · Not your collection"*
- **Read-mostly on guest:** Disable add/delete/clear/barcode-add; allow enrich, play, PDF, labels, insights
- **Transfer-ready copy:** Phase 2 adds *"Copy invite link"* CTA after import completes

## Phased delivery

### Phase 0 — Schema only (zero UI, zero prod risk)

- [ ] SQL migrations deployed to Supabase
- [ ] Backfill personal `collections` + `records.collection_id`
- [ ] Verify prod still works on current app (columns ignored)
- [ ] Log in CHANGELOG

### Phase 1 — Guest crates (you demo)

- [ ] `collections.ts` + `useActiveCollection`
- [ ] Crate switcher + guest banner
- [ ] Import → guest crate (1k cap, 5 crate cap)
- [ ] Scope fetch/enrich/clear/insights/PDF/labels/play to `collection_id`
- [ ] Guard: manual add / barcode / search-add → personal only
- [ ] Smoke: personal crate identical to today

### Phase 2 — Invite & claim

- [ ] `claim_token` generation + `/c/:slug` route
- [ ] Claim on signup (token validation)
- [ ] Transfer `owner_user_id`, `kind` → `personal`, remove from importer's guest list
- [ ] Optional: Discogs username match on claim (stretch)

### Phase 3 — Scale (800–1k libraries)

- [ ] Virtualized collection list
- [ ] Enrichment job progress + resume per crate
- [ ] Insights Web Worker or server snapshot if client stalls

## Prod deploy checklist (every phase)

```bash
npm run build
# Smoke on preview:
# 1. Sign in → personal collection unchanged
# 2. Record count matches pre-deploy
# 3. Play one track
# 4. Insights loads
# 5. (Phase 1+) Guest import + switcher
curl -s https://myvinyl-nine.vercel.app/api/health
npx vercel --prod
```

**Rollback:** Phase 0 is backward-compatible. Phase 1 rollback = redeploy previous bundle; guest rows remain in DB but hidden until re-deploy.

## Open decisions (defaults chosen)

| Question | Default |
|----------|---------|
| After claim, keep copy on your account? | **No** — single canonical copy transfers |
| Preview without login? | Phase 2 — read-only teaser, full view after signup |
| Enrichment on guest before claim? | **Yes** — travels with collection |
| Friend re-sync Discogs post-claim? | Phase 2+ — same username on their account |
